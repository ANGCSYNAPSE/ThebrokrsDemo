VALUE ARCHITECTURE

Never mix money, credits, points, vouchers, or assets

Each type has different ownership, expiry, withdrawal, accounting, and reversal behavior. The interface must label them independently.

MoneyPayment and refund through regulated provider. Not a loyalty balance.
Service CreditRupee-denominated promotional value, restricted to eligible services.
Reward PointsNon-cash loyalty unit with pending, vested, redeemed, expired and reversed states.
Voucher / DiscountSingle-use entitlement or pre-payment price adjustment with explicit terms.
Asset ValueInformational portfolio, land, property or project value—never shown as spendable balance.
ADMIN CONTROL PLANE

How admins operate the marketplace end to end

Every event enters a role-based work queue. Sensitive actions use maker-checker approval, produce compensating events rather than history edits, and notify affected users or partners.

01 · INTAKE

Event or request arrives

Customer application, partner submission, payment event, redemption, fraud alert, or support ticket.

System assigns type, priority, SLA and ownership.
02 · ROUTE

Role-based work queue

Catalog, operations, KYC, finance, rewards, risk, support, or grievance.

No shared super-admin inbox.
03 · REVIEW

360° context

Customer, partner, service, documents, payment, reward, referral and prior support history.

Sensitive data is masked by role.
04 · ACT

Permitted operation

Approve, request changes, assign, refund, pause, compensate, escalate, or resolve.

Reason and evidence required.
05 · CONTROL

Maker-checker

High-risk changes wait for a second authorized reviewer.

Rewards, refunds, KYC override, bank and role changes.
06 · EXECUTE

Event + notification

Domain state changes, projections refresh, and user/partner receives the next action.

Idempotent and auditable.
07 · CLOSE

Resolve or escalate

SLA stops only at a valid terminal state; unresolved work is reassigned or escalated.

Audit trail is immutable.

Core service flywheel

Discover→Trust→Transact→Track→Return

Every vertical uses the same predictable transaction shell while keeping service-specific comparison and tracking.

Referral growth flywheel

Invite→Join→First action→Vested points↺

Rewards vest only after a qualifying action, making acquisition transparent and reducing abuse.

Partner value flywheel

Offer→Activate→Verify→Save + earn↺

Timed codes and bill verification close the partner loop while feeding useful rewards back into Home.

KEY NAVIGATION RULE

Home, Search, Rewards, Offers and Profile remain the five stable destinations. Detailed tasks return to their parent destination.

KYC RULE

Verification appears only when the chosen action requires it, then resumes the abandoned transaction at the exact prior step.

REWARD RULE

Every point change writes to an auditable ledger with pending, vested, reversed and redeemed states.

FAILURE RULE

Payment, KYC and partner-verification failures retain user input and always provide a clear retry or assisted-support path.
